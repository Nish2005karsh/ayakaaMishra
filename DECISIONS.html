Folder Structure
Rule: State management files (BLoC / Provider) live in global bloc/ and provider/ folders — never inside a screen folder. Screen folders contain only *_screen.dart UI files.
lib/ ├── const/ api_routes, app_colors, app_constants, app_session ├── model/ driver_model, vehicle_model, document_model, trip_model ├── data/ │ ├── remote/ dio_client.dart (singleton) │ └── repository/ auth_, vehicle_, document_, trip_repository.dart ├── bloc/ ← GLOBAL state management (BLoC) │ ├── auth/ auth_bloc, auth_event, auth_state │ ├── document/ document_bloc, document_event, document_state │ └── trip/ trip_bloc, trip_event, trip_state (Phase 5) ├── provider/ ← GLOBAL state management (Provider) │ ├── profile_provider.dart │ ├── vehicle_provider.dart │ └── availability_provider.dart ├── screen/ ← UI only, no business logic │ ├── splash/ login/ otp/ dashboard/ │ ├── profile/ vehicle/ documents/ trips/ settings/ └── widget/ custom_button, custom_text_field, otp_input_field, …

company_id Strategy
Source: driver.driver_company from the OTP verify response. Stored in SharedPreferences via AppSession.saveSession() immediately after login. All repository classes read AppSession.companyId — fallback value is 5.

For the team: When backend makes company_id fully dynamic per API call, change only AppSession.saveSession(). No other files need to change. 
// All repositories use this single pattern:
final companyId = AppSession.companyId;  // reads from SharedPreferences

// AppSession.companyId returns:
//   stored value (from driver.driver_company on login)  ← normal case
//   fallback: 5                                          ← before first login
