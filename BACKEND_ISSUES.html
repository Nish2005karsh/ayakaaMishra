Note to backend developer

The app team has reviewed the API documentation and corrected all parameter names and request formats on the client side (driver_id instead of d_id, vehicle_id instead of v_id, FormData for uploads, etc.). The issues in this document are the two remaining problems that can only be fixed server-side.

Everything else (trip_list, upload_docs, vehicle_types, documentNames) is either working or waiting to be re-tested after this backend fix.

Issues resolved on the app side (no backend action needed): trip_list now sends driver_id + tripHistory:false · update_vehicleState now sends vehicle_id (not v_id) · upload_docs now uses FormData with driver_id, document_id, and fields[name]=value array notation.
Remaining Backend Issues
1
vehicle_insert — does not return vehicle_id in any response
POST /vehicle_insert
Critical — Blocks Online/Offline toggle permanently

After a driver registers a vehicle, the app needs to store the vehicle's database ID (vehicle_id) so it can be passed to /update_vehicleState later. Neither the success response nor the "already registered" response currently includes this value. As a result, the Online/Offline toggle is broken for all drivers.

Current response — success case
{
  "code": "0",
  "message": "Vehicle Registered"
}

⚠ No vehicle_id returned.
⚠ Flat format (other endpoints
   use {"status":{...},"data":{...}})
Current response — already registered case
{
  "code": "1",
  "message": "Driver Already registered
    With Another vehicle!!"
}

⚠ No vehicle_id returned here either.
⚠ App has no way to retrieve it.
Expected response — success case
{
  "status": {
    "code": "0",
    "message": "Vehicle Registered Successfully"
  },
  "data": {
    "vehicle_id": 42   ← ADD THIS
  }
}
Expected response — already registered case
{
  "status": {
    "code": "1",
    "message": "Vehicle already registered"
  },
  "data": {
    "vehicle_id": 42   ← ADD THIS TOO
  }
}

What we need from you

  1. Return vehicle_id (the DB primary key from corporate_wheels_vehicles) in both the success response and the "already registered" error response.
  2. Optionally: standardise the response format to match all other endpoints — {"status":{"code":"X","message":"..."},"data":{...}} instead of the current flat {"code":"X","message":"..."}.
2
update_vehicleState — still returns "Missing required parameters" after fix
GET /update_vehicleState
Critical — depends on Issue #1 being fixed first

The app now sends the correct parameter name (vehicle_id, not v_id). However, because vehicle_id was never returned by /vehicle_insert, the app currently has no value to send. The endpoint will work correctly as soon as Issue #1 is fixed and the app receives a real vehicle_id.

What the app will send (once vehicle_id is available)
GET /update_vehicleState
  ?company_id=9
  &vehicle_id=42        ← will be the real ID from vehicle_insert
  &vehicle_status=0     ← 0=ONLINE, 1=OFFLINE

What we need from you

  1. Fix Issue #1 first. Once vehicle_id is returned by /vehicle_insert, the app will store it automatically and send it here. No further changes to this endpoint should be needed.
  2. If you want to test before Issue #1 is fixed: please share what vehicle_id value the driver with driver_id=2 currently has in the database, and we can hardcode it temporarily to confirm the endpoint works.
